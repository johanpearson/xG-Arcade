# Critical Review — xG Arcade Documentation & Setup

Date: 2026-07-07 (fixes applied same day — see status markers below)
Reviewer note: this is a genuine critical pass, not a rubber stamp. Where I
found real issues (verified against actual file contents and, where
relevant, current external facts), they're stated plainly. Where a file is
solid, I say that plainly too — padding this with manufactured criticism
would defeat the point.

> **Status update, 2026-07-07 (same day):** all five concretely-actionable
> findings below were fixed and are marked `✅ FIXED` at the relevant
> point. One finding (backup procedure "duplication") turned out to be
> inaccurate on closer inspection while implementing fixes — corrected in
> place and marked `⚠️ CORRECTION` rather than silently removed, since the
> point of this document is to be trustworthy on a later re-read, not to
> look clean. Findings that are inherently unverifiable until real code
> exists (the seven agents, coding guidelines, contrast values) are
> unchanged — they weren't fixable today regardless.

**Recommendation categories:** **Keep** (fine as-is) / **Update** (real
issue, worth fixing) / **Remove** (not earning its place).

**Overall verdict up front:** the documentation set is unusually thorough
for a pre-code project and internally consistent on the big structural
decisions. The real issues found are: (1) requirement ordering drift in
one section, (2) stale Bicep API versions across all three modules, (3) a
few smaller gaps. Nothing found undermines the architecture itself.

## `docs/` — core documents

### `docs/requirements-document.md`
- **Good:** The Given/When/Then discipline is applied consistently across
  40+ requirements. The separation of "sensible technical defaults" (§5)
  from "genuine open questions" (§6/§7) is a real, valuable distinction
  that's actually been respected throughout, not just stated once and
  ignored.
- **Could be better:** ✅ **FIXED (2026-07-07).** Previously: REQ-204,
  REQ-205, and REQ-206 (live uniqueness, score locking, total score)
  physically appeared *after* REQ-207 through REQ-210 in the document,
  even though they're numbered lower — an artifact of REQ-207-210 being
  inserted at a point in the conversation before 204-206 and never moved.
  Fixed by moving the text (not renumbering) so section 4.2 now reads in
  ascending REQ order. Verified directly after the fix.
- **Could be better:** The REQ number "blocks" (1xx/2xx/3xx/4xx/5xx/7xx/8xx)
  don't monotonically match section order — §4.10 (REQ-710/711, account
  rights) sits after §4.9 (REQ-801-804, testability) even though 710 < 801.
  This reflects the chronological order features were added in, not a
  clean information architecture. Not urgent, but worth knowing it's there.
- **Not good:** None found beyond the above — the content itself (every
  individual requirement) held up under scrutiny.
- **Recommendation:** ✅ Fixed. The section-vs-number-block mismatch
  (§4.10 after §4.9 despite lower REQ numbers) is lower priority and left
  as a known, documented quirk rather than reordered — reordering that
  would be more disruptive for less benefit.

### `docs/architecture-document.md`
- **Good:** Component numbering (COMP-01 through COMP-10) is clean,
  sequential, no gaps or duplicates — verified directly. The five boundary
  rules are specific and enforceable, not vague principles. The data flow
  sections (§6) are detailed enough to actually guide implementation.
- **Could be better:** Some data flows (6.2 especially, guess submission)

- **Not good:** None found.

### `docs/design-document.md`
- **Good:** The v0.1 → v0.2 pivot (dark broadcast → light/clean) was

  it's more a "still open" than a "not good," but worth surfacing here too.

### `docs/implementation-document.md`
